# Список запрещённых калек (межпроектный)

Скилл ведёт межпроектный список калек английских техтерминов, запрещённых в русскоязычном выводе агента. Канонический источник — `~/.claude/calque-blocklist.json`. На список ссылается `~/.claude/CLAUDE.md` (раздел «Russian Communication Quality»).

Всё общение агента с пользователем идёт в одной узкоспециальной тематике — разработка и IT; она (далее «наша тематика») — рамка для всех оценок легитимности и частоты слова в этом скилле.

**Охват — только общие межпроектные кальки.** Доменные терминорешения конкретного проекта (например, avia: `docs/project/terms-variants.md`) сюда не дублируются — у них свой источник истины.

## Главный принцип: добавляй, не оспаривай

Эти правила имеют приоритет над остальным текстом скилла.

1. **Явная кальку запрета авторитетна.** Пользователь дал слово и замену («забань X, по-русски Y») — заносим как есть, в указанной форме. Скилл не отклоняет, не фильтрует, не ставит на паузу. Замеченную коллизию отмечаем одной строкой ПОСЛЕ записи.
2. **Вызов скилла = разрешение на запись.** Отдельного «писать?» скилл не запрашивает.
3. Останавливаемся и спрашиваем ДО записи только при **высокоуровневом** сомнении: неясно, какое слово калька, а какое замена; запрос не разбирается однозначно; операция выходит за рамки списка.

## Рамка оценки коллизии — наша тематика

Легитимность слова и риск «замена испортит обычное употребление» оцениваются **только внутри нашей тематики** — не в русском языке вообще и не в посторонних узких областях.

- Слово, легитимное лишь вне нашей тематики (футбольные «ворота», бытовые «створки»), внутри неё встречается только как калька. Уровень — `substitute`, а не `hint`; понижать `tier` из-за значений вне тематики нельзя.
- Коллизию в отчёте отмечаем как факт, но в рамках тематики. Не обобщаем её на «это частотное слово русского языка» и не предлагаем альтернатив из посторонних областей.
- `detect` не дают лишь словам, частотным в легальном смысле **внутри нашей тематики** (`против`, `вывод`, `живой`). Слово, легальное только снаружи, `detect` получает.

## banned против accepted

Не всякий англицизм запрещается, но решает это **только явное указание пользователя**, не статистика. `accepted` — слова, которые пользователь прямо разрешил агенту (`флоу`, `коммитить`, `пайплайн`, `эндпоинт`, `воркфлоу`); они заносятся командой `--accepted`. Всё, что пользователь даёт на запрет, идёт в `banned` — даже если сам пользователь это слово употребляет.

Почему частота употребления пользователем тут ни при чём: охват инструмента — речь **агента** (что агенту говорить и чего не говорить), а не речь пользователя. «Пользователь употребляет слово» ≠ «пользователь хочет это слово в речи агента»: он может свободно его писать, цитировать, обсуждать — и при этом требовать, чтобы агент его не писал. Поэтому banned/accepted не решается частотой по сообщениям пользователя, и сканирования пользовательских реплик в инструменте нет.

## Сканирование истории — контекст для перевода

`find_calque_uses.py` нужен не для решения banned/accepted, а чтобы перевести термин верно. Частая ошибка — перевести кальку из голого слова: «слип» → «sleep», тогда как в истории это «слип сериализации» (slip, сбой сериализации). Скрипт поднимает, как сам **агент** употреблял термин:

```bash
python3 ~/.claude/skills/calque-blocklist/scripts/find_calque_uses.py <слово>
```

Сканирует только сообщения ассистента по `~/.claude/projects/**/*.jsonl`, группирует вхождения по словоформе и по каждой группе печатает число и до 5 **самых свежих** цитат с контекстом. Свежих — потому что поведение моделей и агента дрейфует; недавние употребления отражают текущее состояние, а не давнее. Вердикта скрипт не выносит: группы и контексты читает агент и сам решает, какая группа — калька и как её перевести, а какая — другое слово с тем же началом основы (калька «слип» против легального «слипаться»). Эта же разбивка по формам показывает, не зацепит ли основа `detect` частое легальное слово.

Что считается употреблением — закрытый список структурных признаков, без угадывания по словам. НЕ употребление (пропускается): код-блок, зона `<meta-discussion>`, инлайн-бэктики (признаки 1-3 переиспользуются из `calque-display.py`) и любое сообщение внутри хода запуска `/calque-blocklist` (по конверту `<command-name>/calque-blocklist`, не по подстроке). Всё прочее — голая проза — это употребление: дисциплина в том, чтобы оборачивать упоминаемую кальку в бэктики, и скрипт это отражает, а не пытается распознать «обсуждение» по словам вроде «калька»/«дословно».

## Контракт данных

`~/.claude/calque-blocklist.json`:

```json
{
  "banned": {
    "<калька>": {
      "replace": ["<замена1>", "<замена2>"],
      "kind": "<вид>",
      "note": "<источник/нюанс>",
      "detect": ["<основа1>", "<основа2>"],
      "tier": "substitute | hint"
    }
  },
  "accepted": ["<принятый жаргон>", "…"]
}
```

`kind` — один из пяти видов (совпадают с §2 `~/.claude/CLAUDE.md`):

| kind | что это |
|---|---|
| `translit` | транслитерация готового русского слова (скоуп, гейт, хоп) |
| `coined-verb` | выдуманный глагол: английский корень + русское окончание (застейджить, маппить) |
| `untranslated` | непереведённое английское/латинское слово в русской фразе (counterexample, view) |
| `pseudo-term` | выдуманный псевдотермин без значения в словаре (условный эталон, блок-нота) |
| `calque-sense` | реальное русское слово в кальковом значении (барьер=gate, вывод=output); запрет только в этом значении |

`replace` — обязателен, непустой; `note` — короткий источник (`scope`, `gate`) или нюанс («запрет только в прозе»). Падежные формы в `replace`/`note` не перечисляем — список читает агент и применяет по смыслу; форму-подсказку можно дать в `note` («формы застейджу, застейджил»).

`detect` и `tier` — опциональные, управляют хуками «Анти-калька» (см. ниже). Без `detect` запись хуками не отслеживается (остаётся только для гейта/ru-check).

## Контроль хуками «Анти-калька» (detect / tier)

Два хука работают по этому списку (ловятся только записи с `detect`); блокировок и перегенерации нет. Распознавание живёт в MessageDisplay: только он видит оригинальный поток до подмены. Прежний Stop-хук `calque-guard.py` ре-читал транскрипт и ловил не финальный, а промежуточный снимок (поток/показ) — то терял `substitute`, то ловил закавыченное, — поэтому распознавание с него снято (файл оставлен заглушкой-no-op).

- **MessageDisplay-хук `calque-display.py`** — единственная точка распознавания. На каждый flush стрима (дельта приходит целыми строками — это оригинал до подмены) в обычной прозе: кальку уровня `substitute` заменяет на экране на верное слово (убирает `скоуп`, ставит «охват», заглавная буква сохраняется) и кладёт в `high`; кальку уровня `hint` на экране оставляет и кладёт в `low`. Находки копятся по сообщению и на финальном flush сливаются в pending по сессии (несколько сообщений одного хода накапливаются). Только экран: транскрипт и контекст модели хранят оригинал. Исключения: блоки кода, `<meta-discussion>`-зоны, инлайн-бэктики, имя «Анти-калька» и сообщения в ответ на просьбу дословной цитаты.
- **PostToolUse-хук `calque-hint-inject.py`** (запасной — UserPromptSubmit) — отдаёт накопленное модели как `additionalContext`, сгруппированное по критичности, и очищает pending. `PostToolUse` срабатывает после каждого вызова инструмента в середине хода, поэтому в длинной цепочке замечание доходит до модели на первом же вызове инструмента после сообщения, а не через ход; `UserPromptSubmit` покрывает ходы без вызовов инструментов. Очередь забирается атомарно — выдаёт ровно одно из двух событий.

Два уровня (`tier`):

- **`tier: "substitute"`** — высокая критичность: MessageDisplay заменяет на экране и кладёт в `high` (additionalContext просит использовать замену впредь). По умолчанию для видов `translit`, `coined-verb`, `pseudo-term`.
- **`tier: "hint"`** — низкая критичность: на экране не трогается, идёт в `low` как «проверь по смыслу». По умолчанию для видов `calque-sense`, `untranslated`.

`tier` нужен, только когда уровень отличается от дефолта вида. Дефолт `hint` для `calque-sense`/`untranslated` держится, лишь пока легальный смысл слова реально встречается в нашей тематике; если внутри неё слово всегда калька — `substitute` (см. «Рамка оценки коллизии»). Примеры: `иммутабелен` — `calque-sense`, но всегда калька, поэтому `substitute`; `ворота` — легально только вне тематики (гол, створки), внутри всегда калька *gate*, поэтому `substitute`; `калька` — `untranslated`, но дублируем её видимо, поэтому тоже `substitute`.

**`detect`** — список основ для ловли (регистронезависимо, по началу слова с открытым окончанием). Правила:
- существительные-транслиты — основа = сама калька (`["скоуп"]` ловит скоуп/скоупа/скоупом);
- глаголы — основа без окончания, но достаточно длинная, чтобы не задеть невинное слово (`матчиться` → `["матчит","матчат"]`, НЕ голое `матч`; `конформить` → `["конформи","конформл"]`, НЕ `конформ`, иначе поймает «конформизм»);
- слишком частотные слова в легальном смысле **внутри нашей тематики** (`против`, `вывод`, `живой`) `detect` не получают — иначе подсказка зашумляется; остаются в списке для гейта; слово, легальное лишь вне тематики, `detect` получает;
- `калька` ловится формами `["калька","калькой","кальке"]`: винительный «кальку» и родит. мн. «калек» опущены намеренно (иначе зацепят «калькуляцию/калькулятор» и «калеки»), а «кальки/калькирование» неразделимы по началу слова — известный пробел покрытия.

Добавляя кальку под видимую замену, **обязательно дай `detect`**; иначе хуки её не увидят. Перед выбором основы проверь её `find_calque_uses.py` и при сомнении прогоном по истории, чтобы основа не ловила невинные слова.

## Запись — ТОЛЬКО через `merge_entry.py`

Ручная правка `.json` запрещена: теряются дедуп, сортировка ключей и проверка инвариантов. Скрипт делает всё это.

Занести кальку:

```bash
python3 ~/.claude/skills/calque-blocklist/scripts/merge_entry.py --json '{
  "тайтл": { "replace": ["заголовок"], "kind": "translit", "note": "title" }
}'
```

Переписать `replace` существующей кальки (вычистить неверные альтернативы, поправить порядок) — используй `replace_set` вместо `replace`:

```bash
python3 ~/.claude/skills/calque-blocklist/scripts/merge_entry.py --json '{
  "коллизия": { "replace_set": ["конфликт", "пересечение"], "kind": "calque-sense" }
}'
```

Для новых лемм `replace_set` и `replace` эквивалентны. Передавать оба одновременно — ошибка.

Занести принятый жаргон (никогда не запрещается):

```bash
python3 ~/.claude/skills/calque-blocklist/scripts/merge_entry.py --accepted "флоу,коммитить"
```

Скрипт: для существующей кальки дополняет `replace` с дедупом (или перезаписывает целиком, если передан `replace_set`); `kind`/`note`/`detect`/`tier` обновляет, если переданы; для новой — создаёт; сортирует ключи `banned`; дедупит и сортирует `accepted`; держит инвариант «слово не может быть и banned, и accepted одновременно»; перезаписывает `.json`. `--dry-run` показывает изменения без записи.

## Валидация

```bash
bash ~/.claude/skills/calque-blocklist/scripts/validate.sh
```

Проверяет JSON, инварианты (replace непуст, kind допустим, нет пересечения banned/accepted, ключи отсортированы). Не зелёное — чинить до объявления готовности.

## Стоп-лист для системного промпта

`~/.claude/CLAUDE.md` импортирует через `@` короткий стоп-лист `~/.claude/calque-stoplist.md` — только самые частые кальки, чтобы напоминание висело в каждой сессии и не раздувало промпт. Это производный артефакт: руками не править, перегенерировать скриптом.

```bash
python3 ~/.claude/skills/calque-blocklist/scripts/render_stoplist.py
```

Скрипт читает канон, считает частоту каждой кальки (по полю `detect`) в собственной прозе ассистента по всем главным транскриптам (`projects/*/*.jsonl`; реплики, обсуждающие кальки, пропускаются по `SKIP_RE` хука, код вырезается), ранжирует по убыванию и оставляет частотные. Пороги: `--min-count` (по умолчанию 20) и `--max` (по умолчанию 15); `--dry-run` печатает без записи. Sense-кальки помечаются «(в кальковом значении)», untranslated — «(латиницей в прозе)»; замены берутся из `replace` канона, поэтому при правке замены через `merge_entry.py` стоп-лист достаточно перегенерировать — он не разойдётся с источником.

## Пере-наполнение по истории (bulk re-seed)

Когда нужно найти кальки, всплывшие с прошлого наполнения:

```bash
python3 ~/.claude/skills/calque-blocklist/scripts/mine_calques.py > /tmp/calque-mining.txt
```

Скрипт сканирует все `~/.claude/projects/**/*.jsonl` (главные сессии + сабагенты + прогоны воркфлоу), вытаскивает окна вокруг маркеров коррекций («кальк», «англицизм», «транслит», стрелки `→`) из реплик пользователя и из ru-check-самоотчётов ассистента, дедуплицирует. Результат — список окон; кальки из них **курируются вручную** (отделить общие от доменных, шум от сигнала), подтверждаются `find_calque_uses.py` и заносятся через `merge_entry.py`. Майнер ничего не пишет на диск, только читает историю.

## Каноническое поведение

Пользователь дал кальку и замену (типовой случай):

1. Определил `kind` по таблице видов.
2. Прогнал `find_calque_uses.py` — прочитал по группам и контекстам, как сам агент употреблял термин, чтобы понять его смысл и перевести верно (а не угадывать из голого слова), и проверить, не зацепит ли основа `detect` легальное слово. Вердикт banned/accepted скрипт не выносит.
3. Записал через `merge_entry.py` сразу → провалидировал `validate.sh`.
4. Отчитался: счётчики (banned/accepted), diff, и одной строкой — замеченные коллизии (факт в рамках нашей тематики, не возражение и не обобщение на язык в целом).
